Console variables (**CVars**) are the engine's primary mechanism for exposing tunable parameters at runtime. They exist as `cvar_t` structs in a hash table maintained by `qcommon/cvar.c` and are accessible from C code, QVM modules, and the developer console.

## The `cvar_t` struct

```c
typedef struct cvar_s {
    char        *name;
    char        *string;          // current value as string
    char        *resetString;     // value from the last Cvar_Get call
    char        *latchedString;   // queued value; applied on next map change
    int          flags;           // CVAR_* flag bits
    qboolean     modified;        // set to qtrue each time the value changes
    int          modificationCount; // incremented on every change
    float        value;           // atof( string )
    int          integer;         // atoi( string )
    struct cvar_s *next;
    struct cvar_s *hashNext;
} cvar_t;
```

`modificationCount` starts at 0 and increments on every change. Code that needs to detect changes (such as `CG_UpdateCvars`) compares a cached copy of `modificationCount` against the current value.

## CVar flags

| Flag | Value | Meaning |
|---|---|---|
| `CVAR_ARCHIVE` | `0x0001` | Written to `q3config.cfg` on exit |
| `CVAR_USERINFO` | `0x0002` | Sent to the server in the userinfo string |
| `CVAR_SERVERINFO` | `0x0004` | Sent to clients in the serverinfo string |
| `CVAR_SYSTEMINFO` | `0x0008` | Sent in the system info configstring |
| `CVAR_INIT` | `0x0010` | Can only be set from the command line |
| `CVAR_LATCH` | `0x0020` | Change takes effect on next map load |
| `CVAR_ROM` | `0x0040` | Read-only; cannot be changed by the user |
| `CVAR_USER_CREATED` | `0x0080` | Created with `set` or `seta` at the console |
| `CVAR_TEMP` | `0x0100` | Not written to config, not archived |
| `CVAR_CHEAT` | `0x0200` | Reset to default when `sv_cheats 0` |
| `CVAR_NORESTART` | `0x0400` | Does not force a map restart when changed |

## Core API

### Creating and reading

```c
cvar_t *Cvar_Get( const char *var_name, const char *value, int flags );
```

If the variable already exists, flags are OR'd in and the function returns the existing pointer. If the value is an empty string, the existing value is not overridden. This makes `Cvar_Get` safe to call multiple times for the same variable.

```c
float  Cvar_VariableValue( const char *var_name );
int    Cvar_VariableIntegerValue( const char *var_name );
char  *Cvar_VariableString( const char *var_name );
void   Cvar_VariableStringBuffer( const char *var_name, char *buffer, int bufsize );
```

### Setting

```c
void Cvar_Set( const char *var_name, const char *value );
void Cvar_SetLatched( const char *var_name, const char *value );
void Cvar_SetValue( const char *var_name, float value );
```

`Cvar_SetLatched` stores the value in `latchedString` rather than applying it immediately. The latched value is applied when the map reloads. `CVAR_LATCH` variables (such as `fs_game`) use this path.

### Modification tracking

The global `cvar_modifiedFlags` integer accumulates the flags of every variable that has changed since the last check. This lets a single comparison detect whether any userinfo or serverinfo variables have changed:

```c
extern int cvar_modifiedFlags;  // qcommon/cvar.c

// Check if any CVAR_SERVERINFO var changed this frame
if ( cvar_modifiedFlags & CVAR_SERVERINFO ) {
    SV_SetConfigstring( CS_SERVERINFO, Cvar_InfoString( CVAR_SERVERINFO ) );
    cvar_modifiedFlags &= ~CVAR_SERVERINFO;
}
```

## vmCvar_t — QVM-side variables

QVM modules cannot hold a pointer to a `cvar_t` because they run in a sandboxed virtual machine. Instead they use `vmCvar_t`:

```c
typedef struct vmCvar_s {
    cvarHandle_t  handle;    // engine-side identifier
    int           modificationCount;
    float         value;
    int           integer;
    char          string[MAX_CVAR_VALUE_STRING];
} vmCvar_t;
```

Registering and updating a QVM CVar:

```c
// Register (called once, usually at module init)
trap_Cvar_Register( &cg_fov, "cg_fov", "90", CVAR_ARCHIVE );

// Update (called each frame to sync latest value into the vmCvar_t)
trap_Cvar_Update( &cg_fov );
```

`CG_RegisterCvars` in `cg_main.c` iterates a static `cvarTable_t` array to register all cgame CVars in one pass. `CG_UpdateCvars` does the same for updating them.

## Important engine CVars

### Renderer

| CVar | Default | Description |
|---|---|---|
| `r_fullscreen` | `1` | Run in fullscreen mode |
| `r_mode` | `3` | Video mode index |
| `r_picmip` | `1` | Texture mip bias (higher = lower quality) |
| `r_vertexLight` | `0` | Use vertex lighting instead of lightmaps |

### Game

| CVar | Default | Description |
|---|---|---|
| `com_maxfps` | `85` | Client frame rate cap |
| `com_blood` | `1` | Enable blood effects |
| `sv_maxclients` | `8` | Maximum connected clients |
| `sv_cheats` | `0` | Allow cheat CVars |
| `g_gravity` | `800` | World gravity |
| `g_speed` | `320` | Player base movement speed |

### Network

| CVar | Default | Description |
|---|---|---|
| `cl_maxpackets` | `30` | Client outgoing packet rate |
| `rate` | `25000` | Client bandwidth (bytes/sec) |
| `snaps` | `20` | Requested snapshot rate |

### Filesystem

| CVar | Default | Description |
|---|---|---|
| `fs_basepath` | `.` | Path to the Quake III installation |
| `fs_homepath` | `~/.q3a` | Write path for user files |
| `fs_cdpath` | `""` | Optional CD-ROM path |
| `fs_game` | `""` | Active mod directory name |
