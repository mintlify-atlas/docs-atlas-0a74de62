The client-side networking layer lives in `client/cl_main.c` (main loop and state management) and `client/cl_parse.c` (inbound message parsing).

## Client state

Three structures hold client state:

| Structure | Description |
|---|---|
| `clientActive_t cl` | Per-connection active state (snapshots, player state, parse entities) |
| `clientConnection_t clc` | Netchan, challenge data, reliable command queues |
| `clientStatic_t cls` | Persistent state across connections (connection state enum, demo info) |

The `refexport_t re` function table (populated by the renderer DLL) is also global in `cl_main.c`.

## Connection sequence

<Steps>
  <Step title="Challenge request">
    `CL_CheckForResend` periodically sends OOB `getchallenge` packets to the server until a `challengeResponse` arrives. The challenge is stored in `clc.challenge`.
  </Step>
  <Step title="Connect packet">
    On receiving the challenge, the client sends an OOB `connect` packet with userinfo and the challenge number. The server responds with a `connectResponse` and establishes a netchan.
  </Step>
  <Step title="Gamestate">
    The server sends a `svc_gamestate` message. `CL_ParseGamestate` processes all configstrings, stores entity baselines, and loads the cgame QVM. The client sends a `new` command to confirm receipt.

    ```c
    void CL_ParseGamestate( msg_t *msg );  // cl_parse.c
    ```
  </Step>
  <Step title="First snapshot">
    Once the cgame is loaded (`CA_PRIMED`) the client begins sending user commands. On the first snapshot the engine transitions to `CA_ACTIVE`.
  </Step>
</Steps>

## Parsing server messages

`CL_ParseServerMessage` (in `cl_parse.c`) reads the opcode byte and dispatches to the appropriate handler.

```c
void CL_ParseServerMessage( msg_t *msg );  // cl_parse.c
```

Opcode dispatch:

| Opcode | Handler |
|---|---|
| `svc_gamestate` | `CL_ParseGamestate` |
| `svc_serverCommand` | `CL_ParseCommandString` → buffered for cgame |
| `svc_snapshot` | `CL_ParseSnapshot` |
| `svc_download` | `CL_ParseDownload` |
| `svc_configstring` | `CL_ParseConfigstring` |

### Snapshot parsing

```c
void CL_ParseSnapshot( msg_t *msg );  // cl_parse.c
```

The snapshot message is parsed into a `clSnapshot_t`. Entity deltas are decoded by `CL_ParsePacketEntities`, which calls `CL_DeltaEntity` for each entity:

```c
void CL_DeltaEntity( msg_t *msg, clSnapshot_t *frame, int newnum,
                     entityState_t *old, qboolean unchanged );
void CL_ParsePacketEntities( msg_t *msg, clSnapshot_t *oldframe,
                             clSnapshot_t *newframe );
```

`MSG_ReadDeltaEntity` reads the bit-level delta from the message and reconstructs the full `entityState_t`.

## User command generation

The client generates a `usercmd_t` every frame from keyboard/mouse input (`CL_CreateCmd` in `cl_input.c`). Commands are buffered in a circular array of `CMD_BACKUP` (64) entries.

```c
void CL_WritePacket( void );  // cl_main.c
```

`CL_WritePacket` sends a `clc_move` message containing up to `cl_packetdup + 1` recent commands (for redundancy against packet loss), delta-compressed with `MSG_WriteDeltaUsercmdKey`.

Key input CVars:

| CVar | Default | Description |
|---|---|---|
| `cl_maxpackets` | `30` | Maximum outgoing packets per second |
| `cl_packetdup` | `1` | Number of previous commands to duplicate in each packet |
| `cl_timeNudge` | `0` | Milliseconds to shift the interpolation window |
| `cl_nodelta` | `0` | Disable delta compression for debugging |

## Command buffer

Console text typed by the player or generated by key bindings is queued in the command buffer before execution:

```c
void Cbuf_AddText( const char *text );   // qcommon/cmd.c
void Cbuf_Execute( void );               // qcommon/cmd.c
```

Commands with no matching local handler and not prefixed with `/` are forwarded to the server as `clc_clientCommand` messages.

## Timeout and reconnect

The client monitors the time since the last valid server packet. If no packet arrives within `cl_timeout` seconds (default 200), the client disconnects.

```c
void CL_CheckTimeout( void );  // cl_main.c
```

On a disconnect the client may attempt to reconnect automatically (`CL_Reconnect_f`) using the last known server address stored in `clc.serverAddress`.

<Note>
The `cl_shownet` CVar (0/1/2) logs incoming packet details to the console, which is useful for diagnosing protocol issues. Level 2 prints every opcode.
</Note>
