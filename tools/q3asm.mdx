`q3asm` is the assembler and linker for the Quake III Virtual Machine (QVM). It reads one or more `.asm` files produced by the LCC compiler and combines them into a single self-contained `.qvm` bytecode file that the engine can load and execute.

Source: `q3asm/q3asm.c`

---

## What q3asm does

q3asm performs a two-pass assembly:

- **Pass 1** â€” resolves all symbol addresses and calculates segment sizes.
- **Pass 2** â€” emits the final bytecode image with all references patched.

It also functions as a **linker**: a `.q3asm` script file lists every `.asm` translation unit to include, and q3asm merges them into one image, resolving cross-file symbol references.

---

## Invocation

```bash
q3asm -f <scriptfile>
```

The `-f` flag points to a `.q3asm` script (without the extension). q3asm reads the script to find the output path and the list of input `.asm` files.

### Example â€” compiling the game QVM

From `code/game/game.sh`:

```bash
q3asm -f ../game
```

This reads `../game.q3asm` (see below) and writes `qagame.qvm`.

## The `.q3asm` linker script

A `.q3asm` file is a plain-text list. The first line specifies the output path with `-o`; every subsequent line is the path to an `.asm` file (without extension) to link in.

## QVM segments

The assembler maintains four named segments (defined in `q3asm.c:128`):

| Segment | Purpose |
|---------|---------|
| `CODESEG` | Executable bytecode instructions |
| `DATASEG` | Initialised 32-bit data (byte-swapped for portability) |
| `LITSEG` | String literals |
| `BSSSEG` | Zero-initialised (BSS) data |

Assembler directives in `.asm` files switch between segments:

```asm
code
  ; instructions go here
data
  ; initialised data
lit
  ; string constants
bss
  ; uninitialised data
```

The complete opcode enumeration is defined in `q3asm.c:35â€“119`. Key categories:

### Control flow

| Opcode | Description |
|--------|-------------|
| `OP_ENTER` | Begin a procedure frame |
| `OP_LEAVE` | End a procedure frame |
| `OP_CALL` | Call a function |

| Opcode | Description |
|--------|-------------|
| `OP_EQ` / `OP_NE` | Equal / not-equal |
| `OP_LTI` / `OP_LEI` / `OP_GTI` / `OP_GEI` | Signed comparisons |
| `OP_LTU` / `OP_LEU` / `OP_GTU` / `OP_GEU` | Unsigned comparisons |

| Opcode | Description |
|--------|-------------|
| `OP_EQF` / `OP_NEF` | Float equal / not-equal |
| `OP_LTF` / `OP_LEF` / `OP_GTF` / `OP_GEF` | Float comparisons |

### Memory access

| Opcode | Description |
|--------|-------------|
| `OP_LOAD1` / `OP_LOAD2` / `OP_LOAD4` | Load 1/2/4-byte value |
| `OP_STORE1` / `OP_STORE2` / `OP_STORE4` | Store 1/2/4-byte value |

| Opcode | Description |

## Executable header

The structure written at the start of a `.qvm` file (`q3asm.c:121â€“127`):

```c
typedef struct {
    int  imageBytes;   // total size after decompression
    int  entryPoint;   // offset of vmMain in CODESEG
    int  stackBase;    // base address of the VM stack
    int  stackSize;    // size of the VM stack
} executableHeader_t;
```

Following the header the four segments are written in order: `CODESEG`, `DATASEG`, `LITSEG`, `BSSSEG`.

---

## Maximum image size

The in-memory image for any single segment is capped at 4 MB (`MAX_IMAGE = 0x400000`, defined in `q3asm.c:136`). Complex mods that exceed this limit must reduce code size or split functionality across multiple QVMs.
