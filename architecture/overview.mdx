Quake III Arena is structured around three broad layers: the **engine core**, the **game modules** (executed in the QVM virtual machine), and the **offline tools** (compiler, assembler, map tools). Understanding which code lives in which layer, and how they communicate, is the key to navigating the source.

## The Three Layers

```
┌──────────────────────────────────────────────────┐
│                  Offline Tools                   │
│   lcc (C compiler)  q3asm  q3map  q3radiant      │
└──────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────┐
│               Game Modules (QVM)                 │
│   qagame.qvm   cgame.qvm   ui.qvm                │
│   (sandboxed bytecode — cannot call libc)        │
└──────────────────┬───────────────────────────────┘
                   │  systemCall / VM_Call
┌──────────────────┴───────────────────────────────┐
│                  Engine Core                     │
│  qcommon  client  server  renderer  sound  botlib│
└──────────────────────────────────────────────────┘
```

### Engine Core (`code/`)

The engine provides all OS-facing functionality: window management, OpenGL, networking, filesystem, sound, and memory management. It is written in C with platform-specific backends in `unix/`, `win32/`, and `macosx/`.

Key modules inside the engine core:

- **qcommon** — shared infrastructure: command buffer, cvars, filesystem, networking, hunk/zone memory, and the QVM loader.
- **client** — window, input, sound, cinematic, and UI glue. Calls `CL_Frame` each tick.
- **server** — game simulation host. Calls `SV_Frame` each tick. Manages connections and snapshot delivery.
- **renderer** — OpenGL scene rendering. Loaded as a DLL / shared object at runtime.
- **botlib** — AAS pathfinding and AI library, used by `qagame.qvm` via the game trap interface.

### Game Modules (QVM)

The three game modules are compiled to `.qvm` bytecode by `lcc` + `q3asm` and loaded at runtime by `VM_Create` (`code/qcommon/vm.c`). They run in an isolated virtual machine and may only interact with the engine through a fixed **trap** interface. This design was chosen for two reasons:

1. **Portability** — a single `.qvm` file runs identically on every platform.
2. **Sandboxing** — game code cannot directly call OS functions, preventing crash-inducing or malicious native calls from untrusted servers.

| Module | File | Loaded by |
|---|---|---|
| `qagame` | `baseq3/vm/qagame.qvm` | Server (`sv_game.c`) |
| `cgame` | `baseq3/vm/cgame.qvm` | Client (`cl_cgame.c`) |
| `ui` | `baseq3/vm/ui.qvm` | Client (`cl_ui.c`) |

### Offline Tools

The tools (`lcc`, `q3asm`, `q3map`, `q3radiant`) are used at content-creation time and are not part of the running game.

## The Frame Loop

The main loop is `Com_Frame` in `code/qcommon/common.c`. Every frame it:

1. Drains the OS event queue via `Com_EventLoop`.
2. Calls `SV_Frame(msec)` — runs the server simulation and dispatches game module calls.
3. Calls a second `Com_EventLoop` — picks up any packets the server generated.
4. Calls `CL_Frame(msec)` — runs client prediction, receives snapshots, renders the scene.

```c
// common.c: Com_Frame (simplified)
void Com_Frame( void ) {
    // ...
    com_frameTime = Com_EventLoop();
    msec = com_frameTime - lastTime;
    msec = Com_ModifyMsec( msec );

    SV_Frame( msec );           // server tick

    Com_EventLoop();            // second pass
    Cbuf_Execute();

    CL_Frame( msec );           // client tick
    // ...
    com_frameNumber++;
}
```

<Note>
On a **dedicated server** the `CL_Frame` call is compiled out (`#ifndef DEDICATED`). On a **listen server** both `SV_Frame` and `CL_Frame` run in the same process.
</Note>

### Server Frame

Inside `SV_Frame`, the server advances the game simulation by calling into `qagame.qvm` via `VM_Call(gvm, GAME_RUN_FRAME, svs.time)`. After the game runs, `SV_SendClientMessages` builds a delta-compressed snapshot for each connected client and transmits it.

### Client Frame

Inside `CL_Frame`, the client:

1. Reads incoming server packets (snapshots, configstrings, server commands).
2. Runs `CG_DrawActiveFrame` inside `cgame.qvm`, which runs movement prediction and assembles the render scene.
3. Calls the renderer to draw the assembled scene.

## The VM Boundary

All communication between the engine and game modules crosses the VM boundary through two mechanisms:

### Engine → Game: `VM_Call`

The engine calls a numbered entry point inside the QVM:

```c
// qcommon.h
int QDECL VM_Call( vm_t *vm, int callNum, ... );

// Example: tell the game module to run one frame
VM_Call( gvm, GAME_RUN_FRAME, svs.time );

// Example: tell cgame to draw the current frame
VM_Call( cgm, CG_DRAW_ACTIVE_FRAME, cl.serverTime,
         stereoView, clc.demoPlayback );
```

The `callNum` values are defined in `g_public.h` (`gameExport_t`) and `cg_public.h` (`cgameExport_t`).

### Game → Engine: System Calls

Game code calls back into the engine through a **negative** call number trapped by the VM:

```c
// The vm_t holds a function pointer to the engine's syscall handler:
int (*systemCall)( int *parms );

// When QVM code executes a syscall, the interpreter/JIT calls:
currentVM->systemCall( args );
```

The system call numbers for each module are defined in their respective `*_public.h` headers (`gameImport_t`, `cgameImport_t`).

## Platform Abstraction Layer

The engine core has no direct OS dependencies. All platform-specific code lives in three directories:

| Directory | Platform |
|---|---|
| `code/unix/` | Linux, FreeBSD — GLX, X11 input, POSIX filesystem, `dlopen` |
| `code/win32/` | Windows — Win32/WGL, DirectInput, `LoadLibrary` |
| `code/macosx/` | macOS — AGL, Carbon input, `dlopen` |

The `code/null/` directory provides stub implementations for all client-side platform code when building a dedicated server with `DEDICATED` defined.

## Data Flow: Input to Render

```
OS input (keyboard / mouse)
        │
        ▼
  Com_EventLoop          ← SE_KEY, SE_MOUSE events
        │
        ▼
   CL_KeyEvent /          ← queued into usercmd_t
   CL_MouseEvent
        │
        ▼
  cl_input.c:            builds usercmd_t
  CL_CreateCmd
        │
        ├──► sent to server (clc_move packet)
        │
        ▼
  SV_Frame:
    VM_Call(GAME_CLIENT_THINK)
    VM_Call(GAME_RUN_FRAME)
    SV_BuildClientSnapshot    ← delta-compressed snapshot
        │
        ▼
  cl_parse.c:            receive snapshot (svc_snapshot)
        │
        ▼
  cgame: CG_DrawActiveFrame
    CG_PredictPlayerState  ← local prediction (bg_pmove.c)
    CG_AddPacketEntities   ← interpolate other entities
    trap_R_RenderScene     ← submit to renderer
        │
        ▼
  renderer: RenderScene → OpenGL draw calls → display
```
